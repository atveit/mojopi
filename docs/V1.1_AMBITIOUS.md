# v1.1 Ambitious Roadmap — Beyond the Port

With the pi-mono port closed at v1.0, three features push mojopi into territory
no local coding agent has shipped. They are independent, parallelizable, and
land in v1.1.

**All three ship in v1.1.0.** Each is carried by one agent in a three-way
parallel dispatch; file ownership is non-overlapping so no conflicts arise.

---

## Feature 1 — Semantic episodic memory

### What it is

A local, vector-indexed memory of past sessions. At each turn the agent
retrieves the top-k most relevant facts about the user, the project, and
past decisions, and injects them into the system prompt. After each session
ends, an extraction pipeline asks the LLM to distill durable facts from the
turn-by-turn log and persists them with timestamps, confidence, and type.

### Why it's novel

Claude Memory, ChatGPT Memory, Cursor — every commercial agent does this
server-side. No local coding agent does it. mojopi gets it with:

- embeddings from `mlx-lm` (Metal), or a simple bag-of-words fallback so tests
  pass without model weights
- numpy cosine similarity for retrieval (no FAISS needed for hundreds of facts)
- JSONL append-only storage at `~/.pi/memory/memory.jsonl`
- LLM-driven extraction that runs post-session to compress raw transcripts
  into 1–5 durable facts

### Architecture

```
src/coding_agent/memory/
├── __init__.mojo           # package marker
├── store.py                # MemoryEntry dataclass + JSONL store + cache
├── embeddings.py           # Metal embedding via mlx-lm (+ bow fallback)
├── extractor.py            # LLM-driven fact extraction from sessions
├── retriever.py            # top-k cosine similarity retrieval
└── memory.mojo             # Mojo thin wrapper
```

### Entry schema

```json
{
  "id": "mem_20260420_001",
  "text": "The user prefers explicit error messages over silent fallbacks.",
  "embedding": [0.12, -0.34, ...],
  "timestamp": "2026-04-20T14:30:00Z",
  "source": "session:a1b2c3",
  "type": "user_preference",
  "confidence": 0.85
}
```

### Public API

```python
from coding_agent.memory import (
    store_memory,        # add an entry
    retrieve_relevant,   # top-k for a query
    forget_memory,       # remove by id
    list_memories,       # all or by type
    extract_from_session,# LLM extraction
    format_for_prompt,   # render for system-prompt injection
)
```

### Slash commands (wired separately after agent completes)

- `/memory` — list all memories
- `/memory add "..."` — add user-supplied fact
- `/memory forget <id>` — remove
- `/memory extract` — run extraction on current session

### Success criteria

- 10+ fast tests pass without model weights (bag-of-words fallback exercised)
- Retrieval returns deterministic top-k for a simple fixture
- Extraction produces valid JSON entries when a mock LLM is stubbed in

## Feature 2 — Speculative decoding with draft model

### What it is

mlx-lm's `stream_generate()` accepts a `draft_model=` kwarg. Pairing a small
draft (e.g. Llama-3.2-1B) with the main model (Llama-3.1-8B) gives the
standard speculative-decoding speedup: the draft proposes several tokens,
the main verifies them in a single forward pass. Same quality, 1.5–2× faster.

### Why it's novel

No local coding agent benchmarks speculative decoding in-context. The
mlx-lm examples are chatbot-style. mojopi measures it on a ReAct-loop
workload (long prompts, short completions) and contributes back the numbers.

### Architecture

```
src/max_brain/
├── speculative.py          # generate_speculative, stream_speculative, benchmark
├── speculative.mojo        # Mojo wrapper
```

### Public API

```python
def generate_speculative(
    prompt: str,
    main_repo: str,
    draft_repo: str = "mlx-community/Llama-3.2-1B-Instruct-4bit",
    max_new_tokens: int = 512,
) -> str

def stream_speculative(prompt, main_repo, draft_repo, max_new_tokens):
    """Yield tokens as they arrive."""

def benchmark_speculative(
    prompt: str,
    main_repo: str,
    draft_repo: str,
    max_new_tokens: int = 64,
) -> dict
    """Return {"speedup": float, "baseline_tok_s": ..., "speculative_tok_s": ...}"""
```

### Success criteria

- Module imports cleanly without model weights
- Fallback: when draft unavailable, falls through to regular `generate_mlx`
- `benchmark_speculative` returns a well-formed dict in dry-run mode
- 6+ tests pass without model weights

## Feature 3 — KV cache persistence across sessions

### What it is

mlx-lm has `make_prompt_cache(model)` that returns the per-layer KV-cache.
MLX's `mx.save` / `mx.load` let us serialize those tensors to disk. Right
now every mojopi turn re-tokenizes and re-prefills the entire history —
wasteful once sessions get 20+ turns deep. Persisting the cache across
turns (and across process restarts) makes long session resume effectively
free.

### Why it's novel

vLLM, SGLang, and TensorRT-LLM do KV cache management for *serving*. No
local agent does it for *resumable single-user sessions*. The insight is
that the cache is a user-private artefact naturally scoped to one session;
we can treat it like any other session resource.

### Architecture

```
src/max_brain/
├── kv_cache.py             # save/load/estimate, path derivation
├── kv_cache.mojo           # Mojo wrapper
```

Storage layout:

```
~/.pi/sessions/<uuid>/
├── transcript.jsonl        # existing session store
└── kv_cache/
    ├── cache_000.safetensors
    ├── cache_001.safetensors
    └── meta.json           # {model, token_count, sha256}
```

### Public API

```python
def save_kv_cache(cache, path: str) -> dict
    """Serialize MLX prompt cache to disk. Returns size + byte count."""

def load_kv_cache(path: str, model) -> list
    """Restore and return the cache."""

def cache_path_for_session(session_id: str) -> str
    """Default path derivation under ~/.pi/sessions/<id>/kv_cache/"""

def estimate_cache_size(cache) -> int
    """Bytes estimate from tensor dtypes and shapes."""

def stream_with_cache(prompt, model_repo, cache, max_new_tokens):
    """Generate tokens using an existing cache; updates in place."""
```

### Success criteria

- Path derivation works and is testable without weights
- `estimate_cache_size` accepts a mock cache object
- Round-trip save/load works with mock tensors
- 6+ tests pass without model weights

## Integration plan

### File ownership (no conflicts)

| Agent | Owns | Does NOT touch |
|-------|------|----------------|
| A | `src/coding_agent/memory/*`, `tests/test_memory.py` | mlx_backend.py, main.mojo, loop.mojo, pipeline.py |
| B | `src/max_brain/speculative.*`, `tests/test_speculative.py` | mlx_backend.py, pipeline.py, loop.mojo |
| C | `src/max_brain/kv_cache.*`, `tests/test_kv_cache.py` | mlx_backend.py, pipeline.py, loop.mojo |

### After all three agents finish

1. Lead runs full suite (`pixi run test`).
2. Lead wires `/memory` slash command into `main.mojo` (single-file change).
3. Lead adds a `v1.1.0` entry to `STATUS.md` + `CHANGELOG.md`.
4. Lead bumps `VERSION` in `main.mojo` to `1.1.0`.
5. Commit, `git tag v1.1.0`, push with tag.

### Why these three together

- **Memory** gives the agent context beyond a single session.
- **Speculative decoding** makes every turn faster.
- **KV cache persistence** makes resume free.

Together they close the gap between "local port of pi" and "local agent that
competes with cloud services on the session experience."

### Out of scope for v1.1

- Multi-model swap via speculative execution (two full runs, pick the better) —
  interesting but cost is 2× not 0.5×.
- Metal kernel fusion for ChatML formatting — overkill.
- RAG over user codebase — use the `read`/`grep` tools for now; add when memory
  grows beyond a few hundred entries.
