# v1.3 plan — Mac-native UI + loop improvements

The pi-mono port is functionally done (v1.2). v1.3 pushes into two directions:

1. **Mac-native UX** — menu bar quick-access app, SwiftUI chat window
2. **Loop + DX improvements** — slash commands, parallel tool dispatch,
   Gemma-family support, env loading, download helper, session search,
   real speculative-decoding benchmark

No VS Code extension — Mac is the focus.

## Environment verified (2026-04-20, macOS 26.3, M2 Max)

- Xcode 26.1.1 ✓ (for SwiftUI)
- rumps Python library — needs `pip install rumps` into pixi env
- Gemma 4 e4b model already downloaded (2.3 GB, verified tool-calling)
- 431 tests currently passing

## The 8 tracks (8 parallel agents — non-overlapping file domains)

Each agent ships a standalone module/project. Lead wires into `main.mojo`
and `loop.mojo` at the end. Pattern confirmed across W3/R1/R2/R3/v1.1/v1.2:
zero merge conflicts every run.

### Track A — macOS menu bar app (`rumps`)

**Owns:** `src/coding_agent/ui/menubar/__init__.mojo`, `menubar.py`, `tests/test_menubar.py`

Click the 🤖 in the menu bar →
- "Ask mojopi…" opens a native Cocoa text input
- "Recent sessions…" lists last 5 sessions
- "Settings" submenu: model, max tokens, auto-memory toggle
- "Quit"

Runs `mojopi -p "<prompt>" --mode json` as subprocess, shows answer in a
Cocoa alert with copy button. No window management — just a tray icon.

**Deps:** `rumps` (installed via `pip install rumps` in pixi env).
**Tests:** Menu creation + callback registration without actually presenting the UI.

### Track B — SwiftUI native chat window

**Owns:** `apps/mojopi-mac/` (Xcode project), no Python/Mojo touched.

A single-window SwiftUI app bundle:
- Menu bar icon (complementing Track A)
- Main window: chat list on left, conversation pane, text input at bottom
- `ProcessRunner` spawns `mojopi --mode rpc` as a subprocess and parses
  JSONL-framed RPC for token streaming and tool-call events
- Native macOS: `.appearance(nil)` for system dark/light mode, Command+N
  for new session, Command+K to clear
- Builds with `xcodebuild -project mojopi-mac.xcodeproj -scheme mojopi-mac build`

**Tests:** `xcodebuild test` builds + runs minimal UI snapshot tests.

### Track C — Expanded slash commands

**Owns:** `src/cli/slash_commands.py`, `slash_commands.mojo`, `tests/test_slash_commands.py`

Functional dispatcher for REPL:

```
/model <repo>       switch model mid-session (clears pipeline cache)
/history            show last N turns (with --session support)
/save <path>        export current session as markdown
/fork               branch current session (copy + new id)
/tokens             show estimated prompt token count
/memory list        list stored memories
/memory add "..."   add a manual memory
/memory forget <id> remove memory by id
```

Lead wires into `main.mojo` `_run_interactive()` after agents complete.

### Track D — Parallel tool dispatch in the loop

**Owns:** `src/agent/parallel_loop.py`, `parallel_loop.mojo`, `tests/test_parallel_loop.py`

When `extract_tool_calls` returns multiple calls AND they are all read-only
(`read`, `grep`, `find`, `ls`), dispatch them via the existing
`parallel_dispatch` module (already 9 unit tests). 3× speedup on read-heavy
turns.

Exposes `maybe_parallel_dispatch(calls, dispatch_fn) -> list[ToolResult]`
that the loop can call unconditionally; it internally decides sequential
vs parallel.

### Track E — Gemma channel tag + error messages

**Owns:** `src/agent/thinking.py` (add one regex), `src/max_brain/error_messages.py` (new), `tests/test_error_messages.py`

- Add `<|channel>thought\n...<channel|>` to `thinking.py` strip patterns
  (Gemma 4 family).
- New `error_messages.py` with `friendly_mlx_error(exc) -> str` that turns
  `ValidationError: 1 validation error for MAXModelConfig device_specs...`
  into `Could not load model X on Metal — try --model <smaller> or see
  docs/MODEL_VERIFICATION.md`.

### Track F — Session search

**Owns:** `src/cli/search.py`, `search.mojo`, `tests/test_search.py`

`mojopi search "auth"` — full-text grep across all
`~/.pi/sessions/*/transcript.jsonl`. Returns session_id, turn_index,
and a snippet. Uses Python `re` + JSONL parsing; no index, no dependency.

Wires into `main.mojo` as a new subcommand: if `argv[1] == "search"`,
dispatch to the search module.

### Track G — Model fetch helper + env-loader

**Owns:** `scripts/fetch_model.sh`, `src/cli/env_loader.py`, `tests/test_env_loader.py`

- `scripts/fetch_model.sh gemma-4-e4b-it-4bit` — preflight disk space,
  show download size, then call `python -c "from mlx_lm import load; load('mlx-community/...')"`.
- `src/cli/env_loader.py` — reads `.env` in cwd and `~/.pi/.env`, exports
  to os.environ BEFORE CliArgs defaults take effect. Supports `MOJOPI_MODEL`,
  `MOJOPI_MAX_NEW_TOKENS`, `MOJOPI_AUTO_MEMORY`, `MOJOPI_SESSION` vars.

### Track H — Real speculative-decoding benchmark

**Owns:** `scripts/bench_speculative.py`, `tests/test_bench_speculative.py`

- Downloads `mlx-community/Llama-3.2-1B-Instruct-4bit` as a draft (if not
  cached) — 700 MB.
- Pairs with `mlx-community/Llama-3.2-3B-Instruct-4bit` (already cached)
  as the main model.
- Measures: baseline throughput vs speculative throughput over 3 runs,
  outputs JSON.
- Updates `docs/BENCHMARKS.md` with real speedup numbers (not the current
  same-model control).

## Execution plan

**Parallel wave 1 (agents A–H dispatched at once):**

All 8 agents ship independent modules. Non-overlapping file domains; no
lead intervention needed during their work. Expected duration: 3–15 min
each, longest wins.

**Lead wiring (serial, after all 8 return):**

1. `main.mojo`:
   - REPL: dispatch `/model`, `/history`, `/save`, `/fork`, `/tokens`,
     `/memory*` through Track C module.
   - New `search` subcommand via Track F.
   - Call Track G `env_loader.load_dotenv()` before `parse_args`.
2. `loop.mojo`:
   - After `extract_tool_calls`, pass calls through Track D
     `maybe_parallel_dispatch` instead of the per-call loop.
3. Full test suite (`pixi run test`).
4. Run real-model run-tier tests against Gemma 4.
5. Build SwiftUI app: `cd apps/mojopi-mac && xcodebuild ...`.
6. Update `STATUS.md`, `CHANGELOG.md`, `README.md`, bump `VERSION = "1.3.0"`.
7. Commit, `git tag v1.3.0`, push with tag.

## Success criteria

- +8 modules, +~50 unit tests.
- Menu bar app launches, accepts prompt, shows answer.
- SwiftUI app builds via xcodebuild; launches a window; sends a prompt.
- `/model <repo>` + `/history` + `/save` + `/fork` + `/tokens` +
  `/memory` slash commands work in the REPL.
- Parallel read-only tool dispatch shows measurable speedup in a
  contrived 3×read test.
- Gemma `<|channel>thought<channel|>` prefix stripped from output.
- `mojopi search "term"` finds the term across sessions.
- `scripts/fetch_model.sh` downloads a model with a progress bar.
- `bench_speculative.py` reports actual speedup number with a real 1B draft.
- Full test suite: **~480 tests pass** (up from 431 at v1.2).

## Out of scope for v1.3

- Tauri / Electron desktop app (SwiftUI is enough for Mac)
- Linux + CUDA validation (no hardware)
- Plugin marketplace / registry (needs hosting)
- App Store distribution (needs certs)
- Git auto-commit integration (needs UX design work)

## Why this order

- A + B (UI) can run in full parallel — different codebases entirely.
- C (slash commands), D (parallel dispatch), E (Gemma), F (search), G (env), H (bench) are all pure backend modules with no shared files.
- Lead wiring is the one serial phase; it's small and well-scoped.
